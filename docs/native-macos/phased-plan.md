# Native macOS Open WebUI Phased Plan

## Goal

Build a production-quality native SwiftUI macOS app that covers normal Open WebUI workflows without a browser or WebView. The app starts local-first with Ollama chat, then grows into provider, RAG, workspace, admin, media, automation, and enterprise parity.

## Phase 0: Discovery And Guardrails

- Audit current Open WebUI feature surfaces from `src/routes`, `src/lib/components`, `backend/open_webui/routers`, `backend/open_webui/models`, and `backend/open_webui/config.py`.
- Keep the native app in a separate SwiftPM module so it can be built and tested without disturbing the existing Svelte/FastAPI app.
- Treat OpenAI account or ChatGPT subscription access as unsupported unless official OpenAI docs provide a native-app OAuth flow for model usage. Verified on 2026-06-03: current official API docs use bearer credentials from API keys or short-lived workload identity federation access tokens, and OpenAI Help says ChatGPT subscription billing is separate from API billing.
- Never scrape ChatGPT, reuse browser cookies, or rely on private endpoints.

## Phase 1: MVP Chat

Scope:
- Native `NavigationSplitView` app shell with sidebar, chat detail, composer, settings, and model picker.
- Ollama auto-status against `http://localhost:11434`, including runtime version from `/api/version` and running model count from `/api/ps`.
- Ollama model listing from `/api/tags`.
- Ollama model pull through `/api/pull` and deletion through `/api/delete` where the local Ollama server supports it.
- Streaming chat from `/api/chat`.
- Local persistence for chats and settings.
- Clear provider errors when Ollama is not reachable.
- Repeatable `script/build_and_run.sh` and Codex Run action.

Verification:
- `swift test`
- `swift build`
- `./script/build_and_run.sh --package`
- `./script/build_and_run.sh --validate-package`
- `./script/build_and_run.sh --verify`
- Manual: launch app, refresh models, send a prompt to a local Ollama model, quit/reopen, confirm chat persists.

Current status:
- Ollama runtime status, model listing, streaming chat, model pull, selected-model deletion, and model-management capability gating are implemented in the native model picker.
- Pull progress status is streamed into the UI and the model list refreshes after pull/delete.

## Phase 4: RAG And Knowledge

Scope:
- File import, document parsing, text chunking, embeddings, vector storage, knowledge collections, reindexing, citations, and `#` context insertion.
- Local-first SQLite/vector storage by default, with provider embeddings where configured.
- PDF, Markdown, text, DOCX, XLSX, and web/Youtube loaders where feasible.
- Build on the Phase 2 text attachment path by promoting imported file text into chunked, indexed knowledge collections.

Verification:
- Tests for chunking, indexing, retrieval ranking, citation mapping, and reindexing.
- Manual: import docs, insert collection context, ask cited questions, delete/reindex.

Current status:
- Text knowledge collections can be created from the native sidebar.
- Text/Markdown-like files, selectable PDFs, and existing native notes can be imported into a collection using the active provider's embeddings endpoint.
- Imported text is chunked, embedded, persisted in a local JSON knowledge snapshot, and retrieved with cosine similarity.
- Collections can be deleted from the native sidebar, which removes their documents and chunks from the local snapshot.
- Imported documents are visible under each native sidebar collection, can open a native indexed-text preview, can be renamed with indexed chunk source names updated, and can be deleted individually, removing their chunks while preserving the collection.
- Existing text or PDF documents can be reindexed from a picked file with the same filename, preserving the document ID while replacing chunks and embeddings.
- Knowledge collections, documents, chunks, and embeddings can be exported to and imported from a native JSON bundle for local backup and migration, and individual collections can be shared as portable native JSON bundles that include only that collection's documents/chunks.
- Knowledge collection creation, collection rename, document import/reindex/rename/delete, collection deletion, knowledge JSON import, document preview selection, and single-collection sharing are blocked when the Knowledge feature is disabled; write/import/reindex/delete actions also enforce `knowledge.write` through the native admin directory, with sidebar write controls disabled for users without that grant.
- Knowledge collections now store user/group access grants; the native sidebar, document preview selection, citation opening, and `#collection` retrieval filter collections against the current local admin-directory user or group membership while preserving unmanaged local-user access.
- Knowledge import and retrieval can use a dedicated embedding model from Settings, prefer provider models whose provider-neutral capability metadata indicates embedding support when unset, and preflight the active provider's embedding capability before indexing or `#collection` retrieval.
- Prompts can mention collections with `#collection-slug`; matching chunks are inserted into provider context and saved as message citations.
- Retrieved citations are shown in the transcript, link back to the indexed source document preview with the cited chunk focused, and are included in Markdown/JSON exports.
- Production vector storage, DOCX/XLSX/web/Youtube loaders, OCR for scanned PDFs, richer collection metadata beyond names, inline answer-span citation mapping, and provider-reported embedding capability metadata where APIs expose it remain planned.
